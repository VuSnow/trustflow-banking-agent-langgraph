==================================================
CONTEXT
==================================================

Hệ thống là banking transaction agent theo mô hình verified workflow:

User request
→ Agent hiểu intent
→ Agent resolve thông tin từ read-model database
→ Guardian kiểm tra risk/permission/confirmation
→ User xác nhận
→ Executor gọi external banking API mock
→ Lưu audit trace

Database không phải core banking ledger thật.

Database chỉ dùng để:
1. Lưu read model/lịch sử giao dịch để Text2SQL agent truy vấn.
2. Lưu thông tin cần thiết để agent tạo API payload: accounts, beneficiaries, billers, cards.
3. Lưu action draft, api payload, api call log, audit log.
4. Lưu fraud report và risk signal cho transaction screening.
5. Demo các workflow chuyển tiền, nạp điện thoại, thanh toán hóa đơn, giao dịch thẻ, risk check, báo cáo lừa đảo, transaction screening.

Không cho agent tự "thực hiện giao dịch" bằng cách sửa bảng transactions.
transactions là transaction history/read model.

==================================================
SCRIPT REQUIREMENTS
==================================================

Script phải:
- Dùng Python 3.
- Chỉ dùng standard library: csv, json, uuid, random, datetime, os, pathlib.
- Không dùng pandas.
- Không dùng faker, trừ khi thật sự cần. Ưu tiên hardcode name lists để không phát sinh dependency.
- Dùng random seed cố định:
  - SEED = 42
  - random.seed(SEED)
- Dùng fixed current time:
  - FIXED_NOW = datetime(2026, 6, 1, 12, 0, 0)
  - Không dùng datetime.now().
- Output 16 CSV files + 1 README.md vào folder `./mock_data/`.
- Nếu folder/file đã tồn tại, overwrite safely.
- Tự tạo folder nếu chưa tồn tại.
- Mỗi CSV có header.
- Encoding UTF-8.
- JSON fields phải được serialize bằng `json.dumps(obj, ensure_ascii=False)`.
- CSV phải được ghi bằng `csv.DictWriter`, không tự concat string.
- Tất cả foreign key phải hợp lệ — validate sau khi generate.
- Dữ liệu phải deterministic/reproducible khi chạy lại.
- Không dùng thông tin cá nhân thật.
- Không dùng full card number thật.
- Không dùng số tài khoản/thẻ thật.
- Generate theo thứ tự dependency:
  customers → accounts → cards → beneficiaries → merchants → billers
  → customer_biller_accounts → transaction_categories → transactions
  → action_requests → api_call_logs → audit_logs
  → fraud_reports → reported_accounts → reported_customers → fraud_decisions

==================================================
ID STRATEGY
==================================================

Dùng deterministic UUID:

uuid.uuid5(uuid.NAMESPACE_DNS, f"{table_name}-{index}")

Ví dụ:
- uuid5("customers-1")
- uuid5("accounts-1")
- uuid5("transactions-1")

Script phải giữ lookup dict/list trong memory để đảm bảo FK join được.

TABLE 9: transactions
==================================================

Purpose:
Lịch sử giao dịch/read model. Dùng cho Text2SQL, spending insight, tìm người nhận cũ, tìm giao dịch thẻ/hóa đơn/nạp tiền.

direction               enum            IN | OUT.
transaction_type        enum            BANK_TRANSFER | CARD_PAYMENT | BILL_PAYMENT | PHONE_TOPUP | CASH_WITHDRAWAL | CASH_DEPOSIT | SALARY | FEE | INTEREST | REFUND

merchant_id             UUID|null       Cho CARD_PAYMENT.
biller_id               UUID|null       Cho BILL_PAYMENT.
beneficiary_id          UUID|null       Cho BANK_TRANSFER. PHẢI thuộc cùng cif_no.

- balance_after nullable.
- Populate ít nhất 30-50% transactions gần nhất theo account.
- Không cần exact core ledger.
- Nếu populate thì phải là số hợp lý, không âm, và gần với account balance pattern.
- Không để Text2SQL demo bị toàn null ở các giao dịch gần nhất.

TABLE 10: action_requests
==================================================

Purpose:
Lưu yêu cầu hành động do agent tạo trước khi gọi external API.

Columns:

action_id               UUID        PK.
cif_no                  string      FK → customers.cif_no.
action_type             enum        TRANSFER(40%) | PHONE_TOPUP(15%) | BILL_PAYMENT(20%) | CARD_LOCK(10%) | CARD_UNLOCK(8%) | CARD_LIMIT_CHANGE(7%)
status                  enum        DRAFT(5%) | MISSING_INFO(10%) | PENDING_CONFIRMATION(10%) | PENDING_OTP(5%) | READY_TO_EXECUTE(5%) | EXECUTED(50%) | FAILED(8%) | BLOCKED(7%)
user_text               text
api_name                enum|null   external_transfer_api | external_phone_topup_api | external_bill_payment_api | external_card_service_api
api_payload             JSON string Payload gửi external API. json.dumps().
resolved_entities       JSON string Entity đã extract/resolve. json.dumps().
missing_fields          JSON string List fields thiếu. json.dumps().
risk_score              numeric     0.0 – 1.0.
risk_tier               enum        GREEN | YELLOW | ORANGE | RED
requires_confirmation   boolean
requires_otp            boolean
created_at              timestamp
updated_at              timestamp   >= created_at.

Risk tier mapping:
- GREEN: risk_score < 0.3
- YELLOW: 0.3 <= risk_score < 0.6
- ORANGE: 0.6 <= risk_score < 0.8
- RED: risk_score >= 0.8

Risk distribution:
- 60% GREEN
- 25% YELLOW
- 10% ORANGE
- 5% RED

Business rules:
- risk_tier = RED → status PHẢI là BLOCKED.
- status = MISSING_INFO → missing_fields KHÔNG được rỗng [].
- status = EXECUTED → api_payload PHẢI có đủ required fields cho action_type.
- status = BLOCKED → không có api_call_logs SUCCESS.
- requires_confirmation = true nếu risk_tier != GREEN hoặc amount > 5,000,000.
- requires_otp = true nếu YELLOW/ORANGE hoặc amount > 10,000,000.
- RED thì blocked, không execute.

Required user_text examples:
- "Chuyển cho Minh 2 triệu như tháng trước"
- "Nạp điện thoại 100k cho số lần trước"
- "Thanh toán hóa đơn điện hôm nay"
- "Khóa thẻ tín dụng của tôi"
- "Chuyển 50 triệu cho người nhận mới"
- "Tăng hạn mức thẻ lên 100 triệu"
- "Chuyển tiền thuê nhà tháng này"
- "Mở khóa thẻ Visa"
- "Trả tiền nước tháng này"
- "Nạp 200k cho mẹ"

response_payload    JSON string Response giả từ external API.
http_status         integer     200 | 400 | 403 | 409 | 500
status              enum        SUCCESS | FAILED
created_at          timestamp   PHẢI > action_requests.created_at. +1-5 seconds.

Rules:
- Chỉ tạo api_call_logs cho actions có status = EXECUTED hoặc FAILED.
- EXECUTED action → api_call_logs.status = SUCCESS, http_status = 200.
- FAILED action → api_call_logs.status = FAILED, http_status = 400 hoặc 500.
- BLOCKED action → KHÔNG có api_call_logs vì bị chặn trước khi gọi API.
- MISSING_INFO action → KHÔNG có api_call_logs.
- Expected row count khoảng 65-75, phụ thuộc số EXECUTED/FAILED actions.

Response examples:
SUCCESS:
{"external_reference":"EXTTRF000001","status":"SUCCESS","message":"Transfer completed"}

FAILED:
{"error_code":"INSUFFICIENT_FUNDS","message":"Not enough balance"}

4. "Thanh toán hóa đơn điện hôm nay"
- Nhiều customers có customer_biller_accounts liên kết EVN.
- Có lịch sử BILL_PAYMENT hàng tháng.
- Có action_requests type BILL_PAYMENT.

5. "Nạp điện thoại 100k cho số đã nạp lần trước"
- Có PHONE_TOPUP lặp lại cho cùng phone number.
- counterparty_name = phone number.
- counterparty_account_no = same phone number.
- phone prefix phải thuộc CARRIER_MAP.

- amount > 5x trung bình customer
- chuyển tiền 2h-5h sáng
- người nhận mới, beneficiary_id = null
- 3+ giao dịch nhỏ trong 5 phút
- card payment ở city khác thường
- tương ứng action_requests risk_tier ORANGE/RED.

7. "Tôi bị trừ phí gì tháng này?"
- FEE transactions tháng 5/2026:
  - "Phi duy tri tai khoan" 11,000
  - "Phi SMS Banking" 11,000
  - "Phi thuong nien the" 150,000
  - "Phi chuyen tien ngoai he thong" 5,500 – 33,000

8. "Lương tháng này vào chưa?"
- SALARY transactions direction=IN, ngày 1-5 tháng 5/2026.
- description: "Luong thang 05/2026" hoặc "Luong T05/2026 - {company}".

9. "Tôi muốn báo cáo lừa đảo"
- Có fraud_reports từ nhiều customers.
- Ít nhất 3 reports cho cùng 1 reported_account_no.
- reported_accounts phải có aggregate data đúng.
- fraud_type đa dạng: SCAM_TRANSFER, FAKE_INVESTMENT, IMPERSONATION.

10. "Chuyển tiền cho tài khoản X" nhưng X bị reported
- Ít nhất 3 fraud_decisions có matched_report_count > 0.
- Ít nhất 3 fraud_decisions = BLOCK.
- receiver_account_no match reported_accounts.account_no.
- action_id link đúng tới TRANSFER actions.

11. "Tài khoản này có an toàn không?" (transaction screening)
- reported_accounts có đủ risk_level: LOW, MEDIUM, HIGH, CRITICAL.
- Ít nhất 2 accounts risk_level HIGH/CRITICAL.
- status = BLOCKED cho CRITICAL accounts.

==================================================
VALIDATION FUNCTION
==================================================

Script PHẢI có function `validate_data()` kiểm tra:

FK integrity:
- accounts.cif_no ∈ customers.cif_no
- cards.cif_no ∈ customers.cif_no
- cards.account_no ∈ accounts có cùng cif_no
- beneficiaries.cif_no ∈ customers.cif_no
- customer_biller_accounts.cif_no ∈ customers.cif_no
- customer_biller_accounts.biller_id ∈ billers.biller_id
- transactions.cif_no ∈ customers.cif_no
- transactions.account_no ∈ accounts có cùng cif_no
- transactions.card_id nếu != null ∈ cards có cùng cif_no
- transactions.merchant_id nếu != null ∈ merchants.merchant_id
- transactions.biller_id nếu != null ∈ billers.biller_id
- transactions.beneficiary_id nếu != null ∈ beneficiaries có cùng cif_no
- transactions.category_id ∈ transaction_categories.category_id
- action_requests.cif_no ∈ customers.cif_no
- api_call_logs.action_id ∈ action_requests.action_id
- audit_logs.action_id nếu != null ∈ action_requests.action_id
- audit_logs.cif_no ∈ customers.cif_no
- fraud_reports.reporter_cif_no ∈ customers.cif_no
- fraud_reports.transaction_ref nếu != null ∈ transactions.transaction_ref
- fraud_decisions.action_id ∈ action_requests.action_id (chỉ TRANSFER actions)
- reported_accounts.account_no phải match ít nhất 1 fraud_reports.reported_account_no
- reported_customers.cif_no phải match ít nhất 1 reported_accounts.linked_customer_cif

Business rules:
- Mỗi ACTIVE customer có ít nhất 1 ACTIVE PAYMENT account.
- CARD_PAYMENT transaction có card_id != null.
- CARD_PAYMENT transaction có merchant_id != null.
- BILL_PAYMENT transaction có biller_id != null.
- BANK_TRANSFER OUT có counterparty_account_no, counterparty_bank_code, counterparty_name != null.
- PHONE_TOPUP có counterparty_name là phone number.
- PHONE_TOPUP có counterparty_account_no == counterparty_name.
- PHONE_TOPUP phone prefix nằm trong CARRIER_MAP.
- action.status == MISSING_INFO → missing_fields != [].
- action.risk_tier == RED → action.status == BLOCKED.
- action.status == EXECUTED → có api_call_logs.status == SUCCESS.
- action.status == BLOCKED → không có api_call_logs SUCCESS.
- api_call_logs.created_at > action_requests.created_at.
- audit_logs cùng action_id có created_at tăng dần.
- fraud_reports.status = VERIFIED hoặc SUBMITTED → phải được tính trong reported_accounts.valid_report_count.
- fraud_decisions.decision = BLOCK → action.status nên là BLOCKED.
- reported_accounts.risk_level = CRITICAL → status = BLOCKED.
- reported_accounts.valid_report_count >= 1.

Uniqueness:
- transactions.transaction_ref unique.
- accounts.account_no unique.
- billers.biller_code unique.
- transaction_categories.category_code unique.
- reported_accounts.account_no unique.
- reported_customers.cif_no unique.

JSON validity:
- Mọi JSON string field parse được bằng json.loads().

Nếu bất kỳ check nào fail:
raise AssertionError(f"VALIDATION FAILED: {details}")
